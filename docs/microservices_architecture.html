Proposed application architecture

The proposed architecture would include microservices running in separate containers (Docker). This ensures that new containers can be spawned on demand and thus ensures scalability

The matchmaking service that we just developed would just be one of many services running in such an environment.

Cloud hosting

I propose that a cloud hosting solution be used which will provide the following benefits:

  • Increased reliability: a specialized vendor can provide a much higher level of availability when compared to the classic on-premise infrastructure.
  • Scalability: using a cloud based solution means that the application is much easier to scale when needed.
  • Security: most cloud based providers utilize end-to-end approaches for secure and hardening their infrastructure.

As for the actual vendor, I propose Amazon's EC2 cloud hosting backed up by the load balancer of the same provider.

Communication between microservices

Microservices would communicate synchronously (e.g. via REST) or asynchronously by utilizing a message queue.

API gateway

Client requests would be handled by API gateways before being handed off to the services, which enables the following:

  • Insulates the clients from how the application is partitioned into microservices
  • Provides the optimal API for each client
  • Simplifies the client by moving logic for calling multiple services from the client to API gateway
  • Translates from a “standard” public web-friendly API protocol to whatever protocols are used internally

Supporting services

The supporting services would include the following (the so called ELK stack):

  • Elasticsearch: for searching and analyzing data
  • Logstash: for log ingestion and storage into Elasticsearch
  • Kibana: for visualizing Elasticsearch data
